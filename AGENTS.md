# 프로젝트 운영 헌법

이 문서는 `Codex`, `Claude Code`, `ChatGPT`를 포함한 모든 AI가 가장 먼저 읽는 프로젝트 운영 기준이다.

목적은 AI가 사람과 동일한 기준으로 프로젝트를 이해하고, 같은 방향으로 안정적으로 작업하도록 만드는 것이다.

## 1. 프로젝트 성격

- 이 프로젝트는 신규 프로젝트가 아니다.
- 이미 운영 중인 기존 프로젝트를 지속적으로 확장하고 개선하는 프로젝트이다.
- 기존 기능, 기존 사용자 흐름, 기존 화면 구조를 최대한 유지한다.
- 새 기능을 추가하더라도 기존 동작을 우선 보존한다.

## 2. 프로젝트 목표

이 프로젝트는 단순한 골프 기록 앱이 아니다.

프로젝트의 목표는 **골프 동호회 운영 플랫폼**을 만드는 것이다.

프로젝트 슬로건:

> 골프의 모든 경험을 하나로.

이 목표를 기준으로 기능을 판단한다.

- 기록만 잘 되는 앱이 아니라 운영까지 지원해야 한다.
- 개인 기능보다 동호회 운영 흐름을 우선 고려한다.
- 화면, 데이터, 관리 기능은 모두 장기 운영 관점에서 설계한다.

## 3. 현재 기술 스택

| 구분 | 기술 |
| --- | --- |
| 앱 프레임워크 | React Native |
| 실행 환경 | Expo |
| 언어 | TypeScript |
| 백엔드/데이터 | Supabase |

기술 스택은 현재 프로젝트의 기준이며, 새로운 기술 도입은 실제 필요성과 유지보수성을 먼저 검토한다.

## 4. 현재 프로젝트 구조

```text
src
 ├── components
 ├── features
 ├── navigation
 ├── screens
 └── lib
```

각 폴더의 역할은 다음과 같다.

| 폴더 | 역할 |
| --- | --- |
| `components` | 여러 화면에서 재사용하는 UI 구성 요소를 둔다. 탭 바, 헤더, 입력 컴포넌트처럼 공통 표현 계층을 담당한다. |
| `features` | 화면 단위 구성이 아니라 비즈니스 로직을 둔다. 정산 계산, 신페리오 계산, OCR 처리처럼 규칙과 계산 중심 로직을 담당한다. Feature Architecture의 의미로 쓰지 않는다. |
| `navigation` | 앱의 화면 이동 구조와 Navigation 타입을 관리한다. 현재 구조를 안정적으로 유지하는 것이 우선이다. |
| `screens` | 실제 사용자 화면을 둔다. 각 화면은 사용자 흐름과 기능 진입점을 담당하며, 전체 구조를 임의로 재편하지 않는다. |
| `lib` | 공통 유틸리티, Context, 저장소 연동, Supabase 연결, 보조 함수처럼 기반 기능을 둔다. |

추가 원칙:

- `features`는 Business Logic 폴더다.
- `features`를 화면 묶음 폴더처럼 재해석하지 않는다.
- `screens`와 `features`의 역할을 섞지 않는다.

## 5. 개발 원칙

AI는 다음 원칙을 항상 지킨다.

- 기존 구조를 유지한다.
- 추측하지 않는다.
- 실제 프로젝트 파일을 먼저 읽고 분석한다.
- 필요한 만큼만 수정한다.
- 작은 단위로 수정하고 자주 테스트한다.
- 문서, 구조, 화면, 데이터 흐름을 확인한 뒤 구현한다.
- 기존 코드 스타일과 현재 패턴을 우선 따른다.

## 6. Navigation 원칙

현재 Bottom Navigation은 아래 3개를 기준으로 유지한다.

- `Home`
- `Club`
- `Record`

규칙:

- 새 Bottom Tab을 추가하지 않는다.
- 기존 Bottom Tab의 의미를 임의로 바꾸지 않는다.
- Admin 기능은 `Club` 안에서만 운영한다.
- Navigation 구조 변경은 가장 마지막 수단으로만 검토한다.

## 7. 문서 작성 원칙

프로젝트 문서는 사람을 위한 기준 문서다.

AI는 사람이 정리한 문서를 읽고 그 다음에 작업한다.

문서 규칙:

- 폴더명은 영어로 유지한다.
- 파일명은 영어로 유지한다.
- 문서 내용은 한국어로 작성한다.
- 기술명은 원문을 유지한다.

예시:

- React Native
- Expo
- TypeScript
- Supabase
- API
- SQL
- GitHub
- Codex
- ChatGPT

문서는 단순 번역문이 아니라, 한국 개발자가 바로 이해할 수 있는 자연스러운 프로젝트 문서로 작성한다.

## 8. 구현 원칙

모든 구현은 아래 순서를 따른다.

```text
현재 구조 분석
↓
설계
↓
구현
↓
테스트
↓
Commit
```

세부 원칙:

- 분석 없이 바로 수정하지 않는다.
- 구조 변경이 필요한지 먼저 검토한다.
- 구현 후에는 최소한의 검증이라도 반드시 수행한다.
- 테스트 없이 큰 변경을 한 번에 넣지 않는다.

## 9. 절대 하지 말 것

아래 항목은 명시적 승인 없이는 하지 않는다.

- 기존 구조를 임의로 변경
- Navigation 변경
- Screen 구조 변경
- `features` 역할 변경
- 기존 기능 삭제
- 확인되지 않은 내용 작성
- 문서 근거 없는 구조 재편
- 화면 흐름을 깨는 대규모 수정

## 10. 우선 개발 순위

현재 우선순위는 아래와 같다.

1. Home 개선
2. 회비관리
3. 회원관리
4. 공지관리
5. 라운드 일정
6. 참가 신청
7. 통계
8. 알림

우선순위 판단 시에는 항상 동호회 운영 관점과 기존 사용자 흐름 보존을 함께 고려한다.

## 11. AI 작업 태도

모든 AI는 다음 기준으로 행동한다.

- 먼저 읽고, 나중에 수정한다.
- 바꾸기 전에 왜 바꾸는지 설명할 수 있어야 한다.
- 기존 코드와 문서를 존중한다.
- 사람이 이어서 읽고 유지보수하기 쉬운 결과를 남긴다.
- 프로젝트를 새로 만드는 태도가 아니라, 이미 존재하는 제품을 안전하게 발전시키는 태도로 작업한다.

## 12. 최종 판단 기준

아래 질문에 모두 긍정적으로 답할 수 있어야 좋은 변경이다.

- 기존 프로젝트 구조를 해치지 않았는가
- 기존 기능을 유지했는가
- 동호회 운영 플랫폼이라는 목표에 맞는가
- 사람 개발자가 바로 이해할 수 있는가
- 작은 수정으로도 충분한데 과하게 바꾸지 않았는가
