# 회비 데이터 구조 초안

## 목적

이 문서는 회비 관리 기능을 구현하기 전에 필요한 데이터 구조를 정리하는 초안 문서다.

현재 단계에서는 실제 `SQL`을 작성하지 않고, 어떤 데이터 단위가 필요한지 먼저 정의한다.

## 기본 방향

회비 관리는 아래 두 흐름을 분리해서 저장한다.

1. 회원 회비 상태
2. 동호회 자금 흐름

회원 회비 상태는 “누가 어떤 상태인가”를 관리하고, 자금 흐름은 “돈이 어떻게 들어오고 나갔는가”를 관리한다.

## 1차 데이터 단위

1차 구현에서 필요한 데이터 단위는 아래와 같다.

| 데이터 단위 | 역할 |
| --- | --- |
| 회비 정책 | 동호회별 회비 운영 기준 |
| 회비 회차 | 월납 또는 연납 기준으로 생성되는 관리 단위 |
| 회원 회비 상태 | 특정 회차에서 회원별 납부 상태 |
| 자금 시작 잔액 | 동호회 자금 관리를 시작하는 기준 금액 |
| 입금 내역 | 동호회로 들어온 금액 기록 |
| 지급 내역 | 동호회에서 나간 금액 기록 |

## 회비 정책

동호회마다 하나의 현재 정책을 가진다.

필요한 정보:

- 동호회 식별값
- 회비 방식: `월납`, `연납`
- 기본 금액
- 공통 마감일
- 월납 자동 생성 여부
- 연납 수동 생성 여부
- 회원별 예외 금액 허용 여부
- 면제 허용 여부
- 일반 회원 공개 범위
- 자금 현황 공개 범위

## 회비 회차

회비 회차는 실제 관리 대상이 되는 월 또는 연도 단위다.

필요한 정보:

- 동호회 식별값
- 회비 정책 식별값
- 회차 기준: 월 또는 연도
- 회차 이름
- 생성 방식: 자동 또는 수동
- 생성일
- 마감일
- 상태

월납은 새 달에 자동 생성한다.

연납은 총무가 필요할 때 수동 생성한다.

## 회원 회비 상태

회원 회비 상태는 특정 회차에서 각 회원이 어떤 상태인지 관리한다.

필요한 정보:

- 회차 식별값
- 회원 식별값
- 상태: `완납`, `미납`, `일부납`
- 적용 금액
- 예외 금액 여부
- 제외 여부
- 면제 여부
- 최근 변경일

목록 화면에는 이름, 상태, 금액, 최근 변경일을 우선 표시한다.

예외 금액, 제외 여부, 면제 여부는 상세 화면에서 확인한다.

## 자금 시작 잔액

자금 현황은 동호회 전체 기준의 시작 잔액 1개에서 출발한다.

필요한 정보:

- 동호회 식별값
- 시작 잔액
- 기준일

이후 잔액은 입금과 지급 내역으로 계산한다.

## 입금 내역

입금 내역은 동호회로 들어온 금액을 기록한다.

필요한 정보:

- 동호회 식별값
- 금액
- 항목
- 날짜
- 짧은 메모
- 작성자

입금 항목은 기본 분류를 제공하고, 총무가 필요하면 추가할 수 있다.

## 지급 내역

지급 내역은 동호회에서 나간 금액을 기록한다.

필요한 정보:

- 동호회 식별값
- 금액
- 항목
- 날짜
- 짧은 메모
- 작성자

지급 항목도 기본 분류를 제공하고, 총무가 필요하면 추가할 수 있다.

## 관계 기준

```text
동호회
↓
회비 정책
↓
회비 회차
↓
회원 회비 상태

동호회
↓
자금 시작 잔액
↓
입금 내역 / 지급 내역
```

## 데이터 원칙

- 회비 상태와 자금 흐름은 분리해서 관리한다.
- 입금과 지급은 서로 상계하지 않고 독립 내역으로 기록한다.
- 상태 변경 이력은 1차 범위에서 별도로 보관하지 않는다.
- 입금/지급 수정과 삭제는 허용한다.
- 운영 단순성을 우선한다.

## 개발 전 검토 사항

- 기존 `Supabase` 테이블과 회원 구조를 확인해야 한다.
- 동호회 식별값과 회원 식별값의 실제 컬럼명을 확인해야 한다.
- 권한 정책은 총무/관리자/일반 회원 기준으로 나눠야 한다.
- 월납 자동 생성 방식은 구현 전에 별도 확정이 필요하다.
