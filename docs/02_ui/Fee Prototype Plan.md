# 회비 관리 UI 프로토타입 계획

## 목적

이 문서는 실제 구현 전에 회비 관리 화면을 어떤 순서로 프로토타입할지 정리한다.

프로토타입의 목적은 기능 완성이 아니라 정보 구조와 운영 흐름 검증이다.

## 프로토타입 원칙

- 기존 `Club` 흐름 안에서 확인한다.
- Bottom Navigation은 변경하지 않는다.
- 총무가 실제로 쓰는 화면처럼 구성한다.
- 데이터 연결 없이 더미 데이터로 화면 구조를 확인한다.

## 1차 프로토타입 범위

| 화면 | 목적 |
| --- | --- |
| Club 운영 메뉴 | 회비 관리 진입 위치 확인 |
| 회비 관리 | 2탭 구조 확인 |
| 회비관리 현황 | 자금 흐름 요약 확인 |
| 회비납부 현황 | 회원 납부 상태 목록 확인 |
| 공지 관리 | 운영 메뉴 확장성 확인 |

## 확인 포인트

### Club 운영 메뉴

- `회비 관리`가 운영 기능 안에서 자연스럽게 보이는가
- `회원 관리`와의 관계가 어색하지 않은가
- 관리자/총무 화면으로 이해하기 쉬운가

### 회비 관리 2탭

- `회비관리 현황`과 `회비납부 현황`의 역할이 명확히 나뉘는가
- 돈의 흐름과 회원 상태가 섞여 보이지 않는가
- 기본 진입 탭이 `회비관리 현황`일 때 총무 관점에서 자연스러운가

### 회비관리 현황

- 잔액과 최근 거래를 한눈에 파악할 수 있는가
- 입금/지급 액션이 바로 보이는가

### 회비납부 현황

- 회원 상태를 빠르게 훑을 수 있는가
- `완납`, `일부납`, `미납` 구분이 직관적인가
- 이름, 상태, 금액, 최근 변경일만으로도 충분히 판단 가능한가

## 더미 데이터 기준

- 회비 방식: 월납 또는 연납
- 상태 값: 완납, 일부납, 미납
- 최근 거래: 입금/지급 혼합 5건 내외
- 회원 목록: 8명 내외

## 승인 기준

1. `Club` 안에서 회비 관리 진입이 자연스럽다.
2. 회비 관리 2탭 구조가 역할 분리에 도움이 된다.
3. 총무 기준으로 첫 화면에서 해야 할 일이 분명하게 보인다.
4. 실제 데이터 연결 전에도 정보 구조가 안정적으로 이해된다.
