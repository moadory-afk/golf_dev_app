# 프로젝트 헌법

이 문서는 이 프로젝트에서 가장 먼저 지켜야 할 운영 원칙을 정리합니다.  
기존 규칙은 유지하고, 오늘부터 필요한 기준을 보강합니다.

## 프로젝트 성격

- 이 프로젝트는 신규 프로젝트가 아닙니다.
- 이미 운영 중인 골프 동호회 플랫폼을 계속 확장합니다.
- 기존 기능은 최대한 유지합니다.
- 새 기능은 기존 구조를 해치지 않는 범위에서 추가합니다.

## 프로젝트 목표

이 프로젝트는 단순한 골프 기록 앱이 아닙니다.  
목표는 `골프 동호회 운영 플랫폼`입니다.

슬로건은 다음과 같습니다.

> 골프의 모든 경험을 하나로.

## 기술 스택

- React Native
- Expo
- TypeScript
- Supabase

## 기본 구조

```text
src
 ├── components
 ├── features
 ├── navigation
 ├── screens
 └── lib
```

- `components`는 재사용 가능한 공용 UI를 둡니다.
- `features`는 Feature Architecture가 아니라 업무 로직을 둡니다.
- `navigation`은 화면 이동 규칙을 관리합니다.
- `screens`는 실제 화면 단위를 둡니다.
- `lib`는 데이터 접근, 공통 함수, 상태 유틸을 둡니다.

## 개발 원칙

- 기존 구조를 유지합니다.
- 추측하지 않습니다.
- 실제 프로젝트를 먼저 분석합니다.
- 필요한 만큼만 수정합니다.
- 작은 단위로 바꾸고 자주 테스트합니다.

## Navigation 원칙

Bottom Navigation은 다음 3개를 유지합니다.

- Home
- Club
- Record

새 Bottom Tab은 추가하지 않습니다.

## 문서 원칙

- 폴더명은 영어를 유지합니다.
- 파일명은 영어를 유지합니다.
- 문서 내용은 한국어로 작성합니다.
- 기술명은 원문을 유지합니다.
- 문서는 사람이 먼저 읽고, AI는 그다음 읽습니다.

## 구현 원칙

항상 다음 순서를 지킵니다.

현재 구조 분석  
↓  
설계  
↓  
구현  
↓  
테스트  
↓  
Commit

## 절대 하지 말 것

- 기존 구조를 임의로 변경하지 않습니다.
- Navigation을 바꾸지 않습니다.
- Screen 구조를 바꾸지 않습니다.
- `features`의 역할을 바꾸지 않습니다.
- 기존 기능을 삭제하지 않습니다.
- 확인되지 않은 내용을 문서에 적지 않습니다.
